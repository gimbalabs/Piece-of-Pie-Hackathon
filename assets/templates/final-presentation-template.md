# Final Presentation Template

This template is a practical boilerplate for the final presentation.

Use it to organize the evidence needed for Builder verification first, then add the optional sections for the Cardano Pie, Real User Pie, and Feedback Pie if they apply.

This is a template, not a separate rule document. The official requirements are defined in the rules and verification docs in this repository.[Rules Directory](../../docs/rules)

## Recommended Structure

1. Project overview
2. Builder verification evidence
3. Product walkthrough
4. Optional track evidence
5. Closing links

## Slide 1: Title

- project name
- team name, if applicable
- presenter name
- track or tracks pursued

Track options:

- Builder Pie
- Cardano Pie
- Real User Pie
- Feedback Pie

## Slide 2: Project Identity

- project name
- one-sentence product description
- official public repository link
- deployed public product link
- official X account or primary X posting account
- team members exactly as registered



## Slide 3: What the Product Does

Show the user-facing problem and the product in plain language.

- who the user is
- what the user can do
- what value the user gets
- where payment happens, if applicable



## Slide 4: Live Demo

Show the product working live.

Recommended flow:

- landing page or entry point
- account creation or onboarding, if relevant
- the core user action
- the payment gate
- the delivered result after payment

## Slide 5: How a User Buys the Product

This is especially important for Real User qualification.

Show:

- pricing page, checkout flow, or payment prompt
- what the user is buying
- what happens after payment
- any account, download, unlock, or fulfillment step



## Slide 6: Public Repository Evidence

Show the official repository and visible build history.

- repository URL
- confirmation that the repository is public
- visible commit history across the build period
- main folders or files that matter



## Slides 7-18: Twelve Official Weekly Update Posts

Provide one slide per official weekly update post, or one summary slide with all twelve links if presentation time is tight.

Each post should be clearly linked and easy to verify.

Recommended format per week:

## Slide 19: Builder Verification Summary

Use one summary slide to show that the required Builder evidence is complete.

Checklist:

- live demo completed
- official public repository shown
- deployed public product link shown
- all 12 official weekly update posts linked
- public evidence is verifiable



# Optional Section: Cardano Pie

Only include this section if the project is pursuing the Cardano Pie.

## Cardano Slide A: Mainnet Functionality

Show the real Cardano mainnet functionality.

Examples:

- mainnet transaction links
- deployed contract or policy IDs
- wallet interaction evidence
- public product flows that use Cardano mainnet


# Optional Section: Real User Pie

Only include this section if the project is pursuing the Real User Pie.

## Real User Slide A: Paying User Evidence

Show proof of at least one real paying user.

Examples may include:

- receipt
- invoice
- billing record
- bank transfer record
- app store or platform payout record
- blockchain payment record with enough context


## Real User Slide B: Customer Acquisition Story

Explain how the user was acquired.

Suggested details:

How they found the product:
Why they decided to pay:
What they paid for:

## Presenter Notes

- Keep every link public and easy to open.
- If private financial evidence must be partially redacted, leave enough detail to verify the claim.
- Put the strongest verification evidence directly in the deck, not only in backup notes.
- If pursuing multiple pies, keep Builder evidence first because the other tracks build on that foundation.
