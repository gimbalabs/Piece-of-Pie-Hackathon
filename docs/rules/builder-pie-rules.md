# Builder Pie Rules

## Builder Pie Summary

The Builder Pie rewards consistent public building over the full hackathon period.

Builder Pie allocation:

- 10,000 ADA
- equal split among qualified participants

Cardano mainnet projects should also review the [Cardano Pie Rules](cardano-pie-rules.md). Qualifying Cardano Pie projects receive a share of the Cardano Pie instead of the Builder Pie.

## Builder Requirements

To qualify for the Builder Pie, a project must meet all of the following requirements.

### 1. Build a User-Facing Software Product

The project must be a user-facing software product, such as:

- a SaaS product
- a web application
- a web3 application
- a platform
- a tool with clear user value

The product must include a clear payment gate, such as:

- subscription pricing
- one-time purchase
- usage-based pricing
- another clear paid access model

### 2. Work Solo or as a Team

Projects may be solo or team-based.

### 3. Register Every Team Member

Every contributor to the project must be explicitly listed during registration.

Unnamed or anonymous contributors are not permitted for a qualifying entry.

### 4. One Person, One Project

Each participant may only join one project.

If a person is found to be participating in multiple projects, the affected projects may be disqualified.

### 5. Maintain One Official Public Repository

Each project must have one official public repository.

That repository must:

- be public
- remain public until final payouts are complete
- show visible commit history
- serve as the official public record of the project work

### 6. Start With an Empty Repository

The official repository must start empty at the beginning of the hackathon.

### 7. Post Weekly Progress Tweets

At least one qualifying public tweet is required every week.

Each required weekly tweet must:

- clearly show project progress
- include `#gimbalabs`
- include `#pieceofpie`
- include `#hackathon`
- mention `@gimbalabs`

Missing a required week results in disqualification.

### 8. Make the Product Publicly Accessible

By the end of the event, the product must be publicly usable on its intended platform.

Examples:

- live public web app
- downloadable mobile app
- browser extension
- other public deployment appropriate to the product

Projects that only run locally do not qualify.

### 9. Deliver a Complete Final Presentation

The final presentation must include:

- a live demo of the product
- the official public repository link
- links to all required weekly tweets
- a deployed public product link

## Automatic Disqualification Triggers

The following issues are disqualifying:

- missing a required weekly tweet
- joining more than one project
- failing to provide verifiable public evidence

## Practical Standard

Builder qualification is based on visible, consistent, verifiable participation over time.
